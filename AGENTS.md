## 「AI客户洞察」模块目标
AI客户洞察是深蓝汽车基于客户录音数据构建的智能经营洞察系统。它通过 AI 对 DCC 外呼、门店接待、试乘试驾、报价谈判、战败回访等客户沟通过程进行深度分析，自动识别客户需求、购车顾虑、竞品对比、政策敏感点、成交阻力、销售行为和门店执行问题，帮助总部制定策略、区域定位问题、门店改善执行，形成从客户声音到经营改善的闭环。

## 代码规范
1. 不要随意重构现有代码。
2. 不要修改无关页面。
3. 不要引入不必要的新依赖。
4. 变更必须最小化，优先沿用既有代码风格与目录结构。
5. 所有新增代码需具备明确模块归属（例如 `modules/ai-customer-insight`）。
6. 提交前必须通过 lint、typecheck、build（若项目存在对应命令）。

## 组件规范
1. 组件优先职责单一，避免“超大组件”。
2. 页面组件与通用组件分层：
   - 页面级：仅组织布局与数据编排
   - 通用级：放置可复用 UI/业务组件
3. Props 必须有清晰类型定义与注释。
4. 组件状态优先本地化，跨页面共享状态需走统一状态管理方案。
5. 禁止在组件内写死接口地址、密钥或环境特定常量。

## 类型定义规范
1. 全面使用 TypeScript（若项目采用 TS）。
2. API 入参与出参类型分离定义，命名建议：
   - `XxxRequest`
   - `XxxResponse`
   - `XxxDTO`
3. 避免 `any`；必要时使用 `unknown` 并在边界收敛。
4. 枚举值统一集中管理，避免魔法字符串散落。
5. 洞察领域核心类型建议预留：
   - `CustomerInsightRecord`
   - `ConversationStage`
   - `PurchaseConcernTag`
   - `CompetitorMention`
   - `StoreExecutionIssue`

## Mock 数据规范
1. Mock 仅用于开发与联调，不进入生产构建。
2. Mock 数据目录建议：`mocks/ai-customer-insight/`。
3. 每类场景单独文件，如：
   - `overview.mock.ts`
   - `concern-distribution.mock.ts`
   - `store-diagnosis.mock.ts`
4. Mock 字段必须与正式类型定义一致。
5. Mock 数据需覆盖：空态、正常态、异常态（缺失字段/极值）。
